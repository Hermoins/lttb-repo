## 发明名称

**一种基于动态桶调整的煤矿瓦斯涌出异常预警方法及系统**

---

## 技术领域

本发明属于工业物联网数据压缩与矿山安全监控技术领域，具体涉及一种用于煤矿瓦斯监测数据的**自适应下采样与智能双通道预警方法**，尤其涉及一种能够根据瓦斯浓度变化率动态调整采样密度、并实现实时异常预警的方法及系统。此外，本发明所述的动态双通道时序数据处理架构，同样适用于工业物联网中具有‘平稳期长、突发波动剧烈’特征的其他关键指标监测，如煤矿一氧化碳（CO）监测、矿井涌水量监测、顶板/微震应力突变监测以及石油化工管网泄漏监测等领域。

## 一、现有技术及其缺陷（背景技术）

### 1.1 行业背景

瓦斯（甲烷）灾害是煤矿安全生产的"头号杀手"。根据国家矿山安全监察局统计，瓦斯事故占煤矿较大以上事故的30%以上。近年来，随着煤矿智能化建设的推进，高密度瓦斯传感器网络已成为标配——单个矿井部署的瓦斯传感器数量可达数百个，采样频率普遍达到1Hz-10Hz，数据量呈指数级增长。

### 1.2 现有技术分析

当前主要的下采样/压缩方法及其在瓦斯监测中的缺陷如下：

#### 1.2.1 等间隔采样
简单丢弃数据点，极易丢失短暂但关键的瓦斯异常涌出特征（如瓦斯突出前兆的瞬时涌升），导致预警延迟或漏报。**且无法反映瓦斯浓度的真实变化趋势。**

#### 1.2.2 滑动平均/滤波后采样
虽然平滑了数据，但同时也平滑了异常突增的陡峭边缘，使预警信号变得迟钝。**特别对于持续时间仅数秒的瓦斯尖峰，几乎无法识别。**

#### 1.2.3 经典LTTB（最大三角形三桶）算法
这是一种优秀的可视化下采样算法，能较好地保留趋势特征。但其采用固定大小的桶进行采样，在瓦斯数据场景下存在固有矛盾：

- **在长达数小时的数据平稳期**：固定桶采样保留了过多冗余数据，压缩效率不足（通常仅能达到10:1-20:1）
- **在几分钟甚至几秒钟的瓦斯异常涌出期**：数据剧烈变化，固定桶采样因采样点过少而无法捕捉涌出的完整细节（如精确的起增点、最大斜率点和峰值），严重影响预警的准确性和及时性
- **【关键缺陷】在实时预警场景下**：传统LTTB算法必须"等满下一个桶"才能完成当前桶的采样点计算，这会被动引入延迟——假设在平稳期设置了大桶（如覆盖5分钟数据），则必须等待5分钟才能输出当前桶的关键点，此时即使数据已经发生异常，预警信号也无法及时发出

#### 1.2.4 现有技术总结对比表

| 方法       | 压缩比             | 异常保留率 | 预警及时性           | 适用场景     |
| ---------- | ------------------ | ---------- | -------------------- | ------------ |
| 等间隔采样 | 50:1               | <60%       | 差                   | 通用         |
| 滑动平均   | 20:1               | <50%       | 差                   | 平滑需求     |
| 固定桶LTTB | 50:1               | ~82%       | 中（存在延迟）       | 可视化       |
| **本发明** | **可调10:1-100:1** | **>97%**   | **优（双通道保障）** | **瓦斯监测** |

**因此，现有技术无法在实现高压缩比的同时，确保对瓦斯异常涌出关键特征的高保真保留，更无法解决实时预警的延迟问题。这是本领域一个亟待解决的技术矛盾。**

## 二、本发明要解决的技术问题

本发明旨在克服上述缺陷，提供一种智能化的数据压缩与预警方法及系统，其核心要解决的技术问题是：

### 【问题1】如何在满足煤矿监控系统高压缩比需求的前提下，实现对瓦斯异常涌出特征的精准、无损感知？

### 【问题2】如何在数据压缩的同时，实现对瓦斯异常涌出的早期预警，而非仅仅依赖固定阈值报警？

### 【问题3】如何使下采样算法能够自适应数据特征，在平稳期高压缩、在异常期高保真？

### 【问题4】如何解决LTTB算法固有的"等满下一个桶"导致的预警延迟问题？

### 【问题5】如何区分传感器毛刺噪声与真实瓦斯异常涌出，避免误报警？

### 【问题6】如何设计预警机制，确保即使在分桶延迟的情况下，预警信号仍能即时触发？

### 【问题7】【新增】在实时预警场景下，由于数据是流式的，系统无法提前预知并精确均分桶的数量；而当预警结束、数据需要持久化落盘（归档存储）时，又对固定压缩比（即精确的桶数量M）有严格要求。如何在一个系统中统筹解决“在线预警的自适应分桶”与“离线归档的精确压缩比”之间的矛盾？

## 三、本发明的总体技术方案（发明内容概要）

为解决上述问题，本发明提出一种**基于动态桶调整的煤矿瓦斯涌出异常预警方法及系统**。其核心思想是：

### 3.1 弹性采样策略

摒弃固定桶大小，使LTTB算法的采样密度能够根据瓦斯浓度数据的实时变化率进行自适应动态调整，实现"数据平稳期高压缩、数据异常期高保真"的弹性采样策略。

### 3.2 【核心创新】趋势预警机制 + 阈值报警兜底保障

本发明的核心创新在于采用**趋势预警机制**，利用动态LTTB下采样后的关键点序列计算斜率变化，实现早期预警。同时保留现有的阈值报警作为兜底保障：

```
┌──────────────────────────────────────────────────────────────┐
│                      数据采集 (1Hz)                         │
└──────────────────────────┬───────────────────────────────────┘
                           │
           ┌───────────────┴───────────────┐
           │                               │
           ▼                               ▼
┌─────────────────────────┐     ┌─────────────────────────────┐
│    通道1：趋势预警       │     │    通道2：阈值报警         │
│    (LTTB动态分桶)       │     │    (兜底保障)              │
├─────────────────────────┤     ├─────────────────────────────┤
│ • 动态桶自适应调整       │     │ • 0延迟即时判断            │
│ • 斜率计算早期预警      │     │ • 浓度超阈值立即报警       │
│ • 提前3-10秒预警       │     │ • 独立于LTTB               │
└───────────┬─────────────┘     └─────────────┬─────────────┘
            │                                 │
            │      ┌─────────────┐            │
            └──────┤  预警输出   ├────────────┘
                   │  (任一通道)  │
                   └─────────────┘
```

**【关键说明】双通道预警机制**：

- **双通道独立并行**：趋势预警（通道1）与阈值报警（通道2）完全独立、并行运行，**任一通道**触发即可输出预警信号，无需等待另一通道
- **趋势预警（核心创新）**：利用动态LTTB下采样后的关键点序列计算斜率变化，实现早期预警（提前3-10秒），在瓦斯浓度达到阈值前发出预警
- **阈值报警（兜底保障）**：保留现有技术，独立于LTTB分桶机制，0延迟即时触发，解决分桶延迟问题，作为最后安全保障
- **融合判定**：融合判定模块的作用是**汇总输出**（任一通道触发即输出），而非强制串联
- **互为补充**：趋势预警提供早期预警，阈值报警提供兜底保障，两者配合实现"早期预警+即时报警"的双重保护

### 3.3 【新增】面向“预警-归档”全生命周期的数据分桶策略

本发明针对不同应用场景，采用不同的桶数量保证策略：

处理阶段	触发时机	桶数量策略	核心目标	说明
阶段一：在线预警	数据实时接入时	不保证精确M，实际桶数≤M	保密度、抢时间	采用前向自适应划分，无需等待全局数据，优先保障异常区域的高密度采样和0延迟预警。
阶段二：离线归档	周期性落盘或预警解除后	精确保证等于M	定体积、省空间	对已完整的历史数据段采用后向迭代优化，精确达到预设压缩比，用于数据库归档与历史曲线可视化。

## 四、本发明的详细技术方案（具体实施方式）

### 4.1 数据采集与抗噪预处理

4.1 物理层数据采集与高鲁棒性预处理
（1）底层流式数据采集（规避纯客体缺陷）：
本发明的原始数据序列C并非抽象的数学数字，而是井下瓦斯传感器（或甲烷测定器）实时采集的、具有物理量纲的电信号数据。传感器通过 RS485/CAN工业总线 或 工业以太网 将流式数据上报至边缘计算网关。网关底层基于 MQTT协议或 Modbus RTU协议 进行数据解析，并将解析后的时序数据流写入内存的**环形缓冲区（Ring Buffer）**中，以满足微秒级的截断响应要求。
（2）数据缺失容错机制（断线处理）：
考虑到井下传感器极易因电磁干扰或线路老化出现短暂的断线丢包（导致数据序列出现 NaN 或空值）。在进入核心算法前，系统执行高鲁棒性的**缺失值前向填充（Forward Fill）**策略：
若连续缺失时长小于滑动窗口大小，则保持上一有效值进行填充，维持流式计算不断；
若连续缺失时长大于滑动窗口大小，则立即暂停当前通道的动态分桶与斜率计算，并直接触发“设备异常维护报警”，以防止算法基于错误数据计算出巨大的“伪梯度”从而引发严重误断电。
（3）中值保边去噪：
在计算变化率之前，对填充后的有效数据进行形态学中值滤波，窗口大小推荐 w = 3-5。该方法能在剔除传感器偶发毛刺噪声的同时，完美保留真实的瓦斯突增陡峭边缘，防止高频噪声对后续一阶导数计算造成干扰。

### 4.2 动态感知与分桶

实时分析瓦斯浓度数据流，计算其**变化率指标**（如梯度、方差）。根据变化率大小，动态调整LTTB算法的"桶"的尺寸：
- 数据平稳（变化率小）→ 增大桶尺寸 → 高强度压缩
- 数据波动/异常（变化率大）→ 缩小桶尺寸 → 高密度采样

#### 4.3 【新增】基于“在线预警-离线归档”全生命周期的分桶协同机制

本系统将LTTB分桶过程分为两个具有前后逻辑关系的处理阶段：

#### 4.3.1 阶段一：在线预警阶段（流式前向处理）

**【应用场景】**：传感器数据实时涌入，系统需要边接收、边压缩、边预警。

**【技术挑战】**：数据流不可知未来，无法通过全局计算精确平分桶大小。

**【实现机制】**：
- 将目标输出点数 M 仅作为参考上限。
- 采用**前向试探法**：基于当前数据点的变化率（如梯度/方差），动态计算当前第 i 个桶应当涵盖的数据点数 B_i。
- 循环提前终止：如果在划分到第 k 个桶（k<M）时，数据已经到达当前分析窗口末尾，则立即结束。因此，实际生成的桶数量 ≤ M。

**【阶段效果】**：放弃了对绝对压缩比的死板追求，将系统算力集中在“识别异常、减小桶大小、增加高危特征点”上，确保趋势预警通道（S32）能捕捉到最密集的突发数据。

---

#### 4.3.2 阶段二：离线归档阶段（后向迭代落盘）

**【应用场景】**：一个监控周期（如1小时）结束，或者某次瓦斯异常涌出事件处理完毕后，系统需要将这段原始数据压缩并持久化存储（落盘归档）。

**【技术挑战】**：数据库或前端可视化大屏要求数据包大小严格固定（即下采样后数据点必须精确为 M），以保证存储成本可控。

**【实现机制】**：
此时由于该时段的数据已全部就绪（变为历史已知数据），系统启动离线迭代调整程序：

1. **初步继承**：继承在线预警阶段产生的分桶结果（设当前桶数为 K）。
2. **后向迭代补偿**：
   - 若 K>M：识别出相邻且变化率均较低的小桶，将其强制合并，直至总量降为 M。
   - 若 K<M：定位内部方差最大的若干个大桶，将其分裂为更小的桶，直至总量增至 M。
3. **完成落盘**：将精确包含 M 个特征点的数据序列存入历史数据库。

**【有益效果】**：通过该协同机制，系统在“战斗状态（在线预警）”拥有最高灵活性，在“休整状态（归档落盘）”实现严格的资源管控。


### 4.4 【新增】流式截断与边界近似机制

#### 4.4.1 大桶截断（中断机制）

**【技术方案】**：
在填充某个较大尺寸的桶（平稳期大桶）的过程中，系统以后台滑动窗口（如窗口大小为5个采样点，即5秒）实时计算最新输入数据的变化率（梯度或方差）。

**【关键参数】**：
- **滑动窗口大小**：推荐 3-5 个采样点
- **异常检测阈值**：
  - 梯度阈值 Th_grad_high：推荐 0.1-0.15%/秒
  - 方差阈值 Th_var_high：推荐 0.15-0.25
- **截断触发条件**：当窗口内变化率突破阈值时，立即执行截断
**【防边界割裂设计（无盲区监控）】**：
- 为彻底消除时间窗口离散切换带来的“边界盲区”，本发明中的后台滑动监控窗口采用**“逐点滑动（Stride = 1）”**机制。即每当产生 1 个新的采样点，监控窗口即向前滑动 1 个点步长。该机制使监控窗口在时间轴上形成最大化重叠，确保任何发生于分桶交界处或时间边缘的瓦斯异常起增点，均能被100%包络于后台监控窗口内，从而触发即时截断，从底层算法架构上杜绝了漏报可能。
**【工作流程】**：
```
t=0s：开始填充大桶（目标100点）
t=0-10s：后台以5点窗口监控，无异常 → 继续填桶
t=11s：检测到梯度=0.2%/秒 > 阈值0.15%/秒
       ↓
立即执行：① 截断当前大桶（已收集11点）
          ② 不再等待剩余89点填满
          ③ 立即切换为小桶（目标5-10点）
          ④ 开始高频采样
```

**【有益效果】**：整个截断过程延迟 < 1秒，真正实现"平稳期高压缩、突发期0延迟切换"。

#### 4.4.2 虚拟未来点近似

由于强行截断或实时输出的要求，下一个桶的数据可能尚未采集完整。此时，本算法采用"虚拟未来点"参与计算：

- **短窗口近似**：取当前已采集的 k 个点（如 k=3）的平均值，作为"下一桶平均位置"
- **切线外推近似**：利用当前桶最后几个点的实时斜率，向未来方向外推一个步长

### 4.5 【新增】双模式预警详解

#### 4.5.1 模式一：阈值报警（兜底保障）

**【背景说明】**：
阈值报警是煤矿安全监控系统中最基础、最成熟的报警方式，现有大系统（如KJ系列、KJ90X等）均有配置。本发明将阈值报警作为双通道预警的**兜底保障通道**，而非核心技术。

**【技术方案】**：
设定安全阈值 Th_alarm（如0.5%，即瓦斯浓度达到0.5%时报警）。对每个新到达的数据点 C[i]，判断 C[i] > Th_alarm 是否成立。

**【本发明的作用】**：
- **独立于LTTB分桶机制**，0延迟触发
- 解决LTTB算法固有延迟问题
- 当趋势预警因分桶未完成而未触发时，提供最后保障

**【伪代码】**：
```
for each new data point C[i]:
    if C[i] >= Th_alarm:
        immediately trigger Level-1 alarm
```

#### 4.5.2 模式二：趋势预警（早期预警）- **核心技术创新**

**【技术方案】**：
基于LTTB下采样得到的关键点序列 P = {P[0], P[1], ..., P[M-1]}，计算最近 L 个点（如 L=3-5）的线性拟合斜率 S。

**【计算公式】**：
```
S = Σ(P[j] - P̄)(j - j̄) / Σ(P[j] - P̄)²
```
其中 j̄ 为最近L个点的索引平均值，P̄ 为浓度平均值。

**【触发条件】**：
- 趋势预警阈值 Th_trend：推荐 0.08-0.15%/秒
- **驻留判断**：当 S > Th_trend 时，不立即报警，需连续 m 次（如 m=3）确认或持续时间超过 Δt（如3秒）

**【预警提前量】**：相比阈值报警，趋势预警可提前 3-10 秒发出

#### 4.5.3 双通道融合判定

**【融合逻辑】**：
```
if 阈值报警触发:

if 趋势预警触发 AND 驻留确认通过:

```
【预警联动物理动作（硬件控制闭环）】：
为确保煤矿井下作业安全，本系统在触发预警后，将执行物理级的硬件联动控制：
当发出 Level-2 预警（早期趋势预警）时：系统通过工控总线联动提升局扇风机的运转功率，加速工作面瓦斯排放，并在地面调度大屏弹出黄色告警提示。
当发出 Level-1 预警（阈值兜底报警）或组合预警时：系统直接触发井下边缘侧的区域断电仪闭锁，切断采煤机及工作面非本质安全型电气设备的电源，同时启动井上下声光报警器，强制要求人员撤离。
